# 팀센터(Teamcenter) 소개

## ENOVIA에서 Teamcenter로 전환 가능한가?

### 결론

**네, 충분히 가능합니다.**

ENOVIA(MatrixOne 기반)를 개발하시던 분이라면 PLM의 핵심 개념(BOM, 리비전, 워크플로우, 변경 관리 등)이 잡혀 있기 때문에 도메인 지식 측면에서 매우 유리합니다. 다만, 개발 언어와 환경(IDE, 프레임워크) 측면에서는 **"학습이 필요한 명확한 차이점"**이 존재합니다.

특히 **"Java 백엔드 + 프론트엔드 경험"**이 있으시기 때문에, Teamcenter의 최신 트렌드인 **Active Workspace (AWC)** 쪽으로 진입하시면 기존 강점을 극대화할 수 있습니다.

---

## ENOVIA vs. Teamcenter 개발 환경 비교

개발자 관점에서 가장 크게 체감하게 될 차이점들을 정리합니다.

| 특징 | ENOVIA (3DExperience) | Teamcenter (Siemens) | 개발자 영향 |
|:---|:---|:---|:---|
| **백엔드 언어** | **Java (JPO)** | **C/C++ (ITK)** & Java (SOA) | **가장 큰 진입 장벽.** Teamcenter의 코어 커스터마이징(ITK)은 C/C++ 기반입니다. 하지만 최근엔 외부 연동 및 클라이언트 로직에서 **Java SOA** 비중이 높아졌습니다. |
| **데이터 모델링** | MQL, Spinner | **BMIDE** (Eclipse 기반) | 툴이 다릅니다. ENOVIA의 `type`, `attribute`, `relationship` 개념은 그대로지만, 이를 정의하고 배포하는 툴이 BMIDE라는 Eclipse 플러그인입니다. |
| **프론트엔드** | JSP, Java 기반 웹 | **Active Workspace (AWC)** | **강점 발휘 가능.** 최신 Teamcenter는 **React, Angular 스타일의 JS/HTML5/CSS** 기반인 AWC를 씁니다. "프론트도 한다"는 점이 여기서 빛을 발합니다. |
| **레거시 UI** | PowerWeb 등 | **RAC (Rich Client)** | 구형 Rich Client(RAC)는 **Java Swing/SWT** 기반입니다. Java를 잘하시니 이쪽 유지보수나 개발도 금방 적응하실 겁니다. |
| **IDE** | Eclipse/IntelliJ | **Eclipse (BMIDE)**, VS (C++) | 둘 다 Eclipse 생태계와 친숙합니다. |

## 추천 커리어 패스

### 1. Java SOA & AWC (Active Workspace) 개발

- **이유:** 이미 Java에 능숙하고 프론트엔드 경험이 있으므로, 굳이 처음부터 C++ ITK라는 험난한 길을 깊게 파기보다 **"Java SOA(백엔드 서비스) + AWC(최신 웹 프론트)"** 조합으로 접근하시면 빠르게 적응할 수 있습니다.
- **시장성:** 현재 Teamcenter 시장도 무거운 RAC(설치형 클라이언트)에서 가벼운 웹 기반인 AWC로 넘어가는 추세라 AWC 개발자 수요가 높습니다.

### 2. C++ ITK (Server-side) 학습

- Teamcenter의 아주 깊숙한 코어 로직(데이터 저장 시점의 훅킹 등)을 건드려야 한다면 C/C++ 기반의 ITK(Integration Toolkit)를 피할 수 없습니다.
- 하지만 Java 백엔드를 하셨으니 "로직" 자체는 문제가 안 되고, C++의 포인터나 메모리 관리 같은 문법적 특성만 보완하시면 됩니다.

### 요약

> **"PLM 업무 프로세스는 이미 다 아시니, 도구(Tool)와 언어(Language)만 바꾸면 됩니다. 특히 Java와 프론트엔드 스킬을 살려 'Teamcenter SOA + Active Workspace' 개발자로 포지셔닝하면 ENOVIA 경력을 인정받으면서 성공적으로 전향할 수 있습니다."**

## Active Workspace (AWS)란?

**네, 정확합니다. 'Active Workspace (AWS)'가 맞습니다.**

개발자 관점에서 아키텍처를 조금 더 구체적으로 설명하면, 코어 로직은 여전히 레거시(C++/SOA)가 강력하게 자리 잡고 있지만, 사용자 인터페이스와 비즈니스 로직의 소비 방식은 완전히 **Active Workspace** 중심으로 재편되었습니다.

### 1. Active Workspace (AWS) 정의

- **정의:** Teamcenter의 차세대 **웹 기반 클라이언트(Thin Client)**이자, 모든 Teamcenter 데이터를 접근하는 표준 UI 프레임워크입니다.
- **역할:** 과거의 무거운 데스크톱 클라이언트(RAC - Rich Application Client, Java Swing/SWT 기반)를 대체하고 있습니다. 현재 Siemens의 모든 신규 기능 개발과 로드맵은 RAC가 아닌 Active Workspace에 집중되어 있습니다.
- **특징:** 별도의 설치 없이 브라우저(Chrome, Edge 등)에서 바로 실행되며, 모바일/태블릿 등 디바이스에 구애받지 않는 반응형 웹 디자인을 제공합니다.

### 2. 아키텍처 변화 (C++ to Web)

개발자 관점에서 보면 "C++에서 웹으로 바뀌었다"는 다음과 같은 구조적 변화를 의미합니다.

#### Core (Business Logic Layer)
- 여전히 **C++**로 작성된 방대한 ITK(Integration Toolkit) 함수들이 핵심 엔진(TC Server)을 돌리고 있습니다. 퍼포먼스와 레거시 호환성 때문에 이 부분은 쉽게 바뀌지 않습니다.

#### Service Layer (SOA)
- C++ 코어 위에는 **SOA(Service Oriented Architecture)** 레이어가 감싸고 있습니다. 외부에서는 이 SOA를 통해 코어 로직을 호출합니다.

#### Web Layer (Active Workspace)
- **Frontend:** HTML5, CSS, 그리고 **AngularJS(초기)** / **React(최신 버전으로 이동 중)**와 같은 모던 자바스크립트 프레임워크를 사용합니다. 'Declarative UI'라는 개념을 써서 JSON 설정만으로 UI를 구성하도록 유도합니다.
- **Backend (Microservices):** Active Workspace Client 자체를 구동하기 위한 경량 서버가 있으며, 이는 주로 **Node.js** 또는 Java 기반의 웹 티어에서 돕니다. 이 웹 티어가 TC Core의 SOA를 호출하여 데이터를 가져와 JSON으로 프론트에 뿌려주는 구조입니다.

### 3. 요약 (ENOVIA 개발자와의 비교)

- **ENOVIA(3DEXPERIENCE):** 전통적으로 **Java(JPO)** 기반의 백엔드가 강력하고, 프론트엔드는 위젯(Widget) 기반의 웹으로 넘어왔습니다.
- **Teamcenter:** **C++(ITK)** 기반의 코어 위에, **SOA**를 거쳐, **Active Workspace(Web/JS)**로 서빙하는 구조입니다.

따라서 **"코어는 C++이지만, 사용자 접점(Client)은 Active Workspace라는 웹 환경으로 완전히 넘어갔다"**고 이해하시면 정확합니다.

## Teamcenter 2312 기술 스택

화면 구성과 버전을 보면 개발자들은 뜯어보지 않아도 대략적인 스택을 유추할 수 있습니다.

**Teamcenter 2312 (Active Workspace 6.3.x 버전)** 인터페이스의 핵심 기술 스택은 바로 **'React'**입니다.

하지만 단순히 "리액트로 짰다"라고 하기엔 Siemens만의 독특한 **'Declarative UI(선언형 UI)'** 프레임워크가 아주 두껍게 깔려 있습니다.

### 1. 화면 뒤의 핵심 기술: React + Declarative Framework

과거 Active Workspace(AWS) 초기 버전(1.0~5.x)은 **AngularJS** 기반이었지만, **2312 버전**은 **React**로 완전히 엔진이 교체된 세대입니다.

- **렌더링 엔진:** **React** (Virtual DOM 사용)
- **상태 관리:** **Redux** (Angular의 `$scope`나 `appCtxService`를 대체하여 전역 상태 관리)
- **빌드/번들링:** **Webpack** (최신 버전에서는 Vite 등으로 넘어가는 추세도 보임)

### 2. "코딩"이 아닌 "설정"으로 그리는 화면 (Declarative UI)

이 부분이 ENOVIA 개발자 입장에서 가장 흥미로운(혹은 답답할 수도 있는) 지점입니다. 이 화면을 개발할 때 개발자는 순수 `.jsx`나 `.tsx` 코드를 짜는 게 아니라, **JSON 설정 파일**을 작성합니다.

- **View Model (JSON):** 화면에 데이터가 어떻게 바인딩될지, 버튼을 누르면 어떤 'Action'이 일어날지를 JSON으로 정의합니다.
  - *예: "id가 'submit-btn'인 버튼을 클릭하면 'saveItem'이라는 SOA 서비스를 호출해라."*
- **View (HTML):** 순수 HTML 태그 대신 Siemens가 미리 만들어둔 **Custom Element**(`<aw-button>`, `<aw-list>`, `<aw-command-bar>`)를 배치합니다.
- **작동 원리:** 런타임에 Siemens의 프레임워크가 이 JSON과 HTML을 읽어서, 미리 만들어둔 **React 컴포넌트**로 변환하여 화면에 뿌려줍니다.

### 3. 우측의 3D 뷰어 (Web Visualization)

화면 오른쪽의 하드디스크 3D 모델은 별도의 플러그인이 아닙니다.

- **기술:** **WebGL** 표준을 사용합니다.
- **엔진:** Siemens의 **'Vis Web'** (또는 JT2Go Web)이라는 자체 라이브러리를 사용하며, 내부적으로는 **Three.js**와 유사한 로직으로 JT 포맷(Siemens의 3D 표준 포맷)을 브라우저에서 직접 렌더링합니다.

### 요약

보시는 화면은 **"React 위에서 돌아가지만, 개발자는 JSON 설정(Declarative UI)을 통해 조립한 결과물"**입니다.

- **장점:** 지멘스가 제공하는 컴포넌트(`<aw-list>`, `<aw-chart>`)를 쓰면 개발 속도가 빠르고 UI 통일성이 보장됩니다.
- **단점:** ENOVIA 커스터마이징처럼 자유롭게 DOM을 조작하거나 외부 라이브러리(예: D3.js, Chart.js)를 마음대로 가져다 쓰기가 매우 까다롭습니다(Wrapper를 만들어야 함).

**한 줄 평:** "껍데기는 모던 웹(React)이지만, 속은 지멘스의 철학(Configurable > Customizable)이 강력하게 지배하는 구조"

## 프로젝트 투입 시 주요 업무

JSON으로 UI를 '설정'한다고 하니 "그럼 나는 코딩 안 하고 설정 파일만 만지는 건가?" 하는 우려가 드실 텐데요. 실제 프로젝트에 투입되시면 ENOVIA 개발과는 업무의 결이 조금 다릅니다.

Teamcenter(Active Workspace) 프로젝트에 투입된 개발자가 주로 하게 되는 업무는 크게 세 가지 덩어리로 나뉩니다.

### 1. 프론트엔드 개발 (화면 개발)

가장 눈에 띄는 업무지만, **"순수 코딩보다는 '조립'과 '연결'에 가깝습니다."**

- **JSON 뷰 모델 작성:** 화면 레이아웃, 버튼 위치, 데이터 바인딩을 정의하는 JSON 파일을 만듭니다. (`i18n` 다국어 처리 포함)
- **Action & Event 연결:** "버튼을 누르면 -> SOA 서비스를 호출하고 -> 결과를 팝업으로 띄운다"라는 흐름을 JSON 내의 `actions` 섹션에 정의합니다.
- **Custom Component 개발:** (이게 진짜 개발) 지멘스가 제공하는 기본 컴포넌트만으로는 고객 요구사항(예: 복잡한 그리드, 특수한 차트)을 못 맞출 때가 많습니다. 이때는 **React 컴포넌트**를 직접 코딩해서 Active Workspace에 이식하는 작업을 합니다.

### 2. 백엔드 커스터마이징 (SOA & ITK)

사실 많은 개발 공수는 여기서 발생합니다. 화면이 아무리 예뻐도 데이터가 없으면 껍데기니까요.

- **SOA (Service Oriented Architecture) 개발:** 프론트(Active Workspace)가 호출할 API를 만듭니다. ENOVIA의 JPO 작성과 비슷하다고 보시면 됩니다.
- **ITK (Integration Toolkit) 프로그래밍:** C/C++로 Teamcenter 코어 로직을 건드리는 작업입니다. 데이터 생성 시 자동 채번, 상태 변경 시 유효성 체크(Validation), 타 시스템으로 데이터 전송 같은 무거운 로직은 여전히 C++로 짭니다.
  - *요즘은 ITK 로직을 C++ 대신 Java나 .NET으로 짤 수 있는 래퍼(Wrapper)도 많이 쓰지만, 퍼포먼스가 중요한 대형 사이트는 여전히 C++입니다.*

### 3. 데이터 매핑 및 구성 (BMIDE)

ENOVIA의 'Schema'나 'Type' 정의와 같은 작업입니다.

- **BMIDE (Business Modeler IDE):** 이클립스 기반의 툴을 써서 데이터 모델(Business Object)을 만들고 속성(Property)을 추가합니다.
- Active Workspace 화면에 어떤 속성을 보여줄지 결정하는 것도 여기서 1차적으로 정의합니다. (Runtime Property 등)

### 결론: 프로젝트 투입 시 예상 시나리오

만약 **Java 백엔드와 프론트 경험**이 있으시다면 다음과 같은 업무를 받으실 확률이 높습니다.

1. **"이 데이터 좀 가져와 주세요":** 기존 ITK/SOA를 수정하거나 새로 짜서 프론트로 넘겨줄 데이터를 가공하는 **백엔드 작업**. (Java나 C++ 사용)
2. **"화면 동작이 이상해요":** JSON으로 정의된 Action 흐름(Workflow)이 꼬인 것을 디버깅하거나, **React로 된 커스텀 컴포넌트**를 수정하는 작업.

### 한 줄 요약

"단순 화면 배치는 JSON '노가다'에 가깝지만, **복잡한 비즈니스 로직은 여전히 백엔드(SOA/ITK)에서 코딩**해야 하고, 고객이 **'기본 기능 말고 딴 거 해줘'** 할 때 React 실력이 빛을 발하게 됩니다."
